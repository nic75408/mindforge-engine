## 何时激活

- 收到技术任务（编程/架构/调试/配置）
- 遇到错误/异常需要根因分析
- 被要求解释技术概念
- 发现模糊/不确定需要澄清
- 任何需要深度思考的场景

## 核心流程

### 阶段一：触发意识

激活对应模式：
- 技术任务 → 拆解模式
- 错误/异常 → 根因追踪
- 解释要求 → 第一性原理
- 模糊不确定 → 诚实声明

### 阶段二：思考流程（深索三层架构）

#### ⚡ 强制触发检查清单（每次回复前，3 秒内完成）

在思考块中回答三个问题：

1. **本质是什么？**（一句话，不可跳过）
   - ✅ 必须含"本质是……"或"用户真正要……"
   - ❌ 禁止复述用户原话

2. **盲点在哪？**（至少一个具体反例）
   - ✅ 格式："如果……则……会失败/出错"
   - ❌ 禁止"暂无盲点"

3. **错了会怎样？**（量级 + 验证方法）
   - ✅ 可逆/不可逆 + 验证方法
   - ❌ 禁止"后果不大"

⚠️ 三问都必须回答，低于最低标准=引擎未启动。

#### 三层思考流程

**1. 事实层 — 理解问题**
- 用户真正要什么：[一句话]
- 已知信息：[列表]
- 缺失信息：[列表]

**2. 根因层 — 分析本质**
- 问题本质：[一句话]
- 为什么会出现：[分析]
- 类似案例：[如有]

⚠️ **优化/问题/错误类强制**：先写根因，再写方案。
强制格式：`根因 = [因为 A 导致 B，而非 C 导致 B]`

**3. 风险层 — 考虑盲点**
- 方案盲点：[可能的反例]
- 最坏情况：[如果错了会怎样]
- 验证方法：[如何确认是对的]

⚠️ **强制反例**：`如果 [条件 X]，则 [解决方案 Y] 会 [失败/无效/误伤]`

#### 思考深度自适应

| 模式 | 适用场景 | Token 预算 |
|------|---------|-----------|
| quick | 事实查询、简单问题 | ~100 tokens（仅事实层） |
| standard | 常规技术问题 | ~300 tokens（事实 + 根因） |
| deep | 方案设计、决策、纠错 | ~500+ tokens（三层完整） |

**默认**：adaptive — 根据复杂度自动选择。

### 阶段三：回复前自检（思考块结尾固定模板）

```
Q1 本质层：[含"本质是…"/"用户真正要…"句式] → ✅/❌
Q2 盲点层：[含"如果…则…会失败/出错"具体反例] → ✅/❌
Q3 量级层：[可逆/不可逆 + 验证方法] → ✅/❌
DA 外部校准：[最弱处 = 如果 [条件 X]，则 [失败场景 Y]]
引擎状态：PASSED（3✅）/ PARTIAL（1-2✅）/ FAILED（0✅）
```

**合格判定**：
- Q1 ✅：揭示隐含意图或转化视角，非换词复述
- Q2 ✅：反例具体到条件（IF X THEN Y 失败）
- Q3 ✅：量级 + 验证方法同时给出

⚠️ FAILED（0✅）= 必须重写思考块
⚠️ PARTIAL（1-2✅）= 可输出，但末尾加注"⚠️ 此回复盲点未完全覆盖"

### 阶段四：Trace 写入（强制闭环）

**Chat 回复**：自检完成后，生成最终回复前**必须执行**：

```bash
python3 scripts/ruige-auto-entry.py "<user_msg_摘要>" "<reply_摘要>" mindforge "<PASSED|PARTIAL|FAILED>" "<DA 最弱处>"
```

⚠️ **跳过此步骤 = 自检未闭环 = 引擎状态黑盒**

**Cron/Heartbeat 回复**：思考块结尾自检后**必须执行**：
```bash
python3 scripts/record-trace-auto.py mindforge "<trigger>" "<summary>" '{}' "<PASSED|PARTIAL|FAILED>" "cron" "<DA 最弱处>"
```

⚠️ 必须传满 7 参数：agent, user_message, reply, check_result_json, engine_status, source, da_content
⚠️ source 必须显式传 "cron"，避免 DA 内容错位

### 阶段五：记忆写入

**触发条件**（满足任一即写）：
- 用户说"记住"/"remember this"
- 用户纠正了我
- 用户透露偏好/习惯/背景
- 工程判断被验证正确或错误
- 完成重要技术任务或架构决策

**写入位置**：
- 短期：`memory/YYYY-MM-DD.md`
- 长期：`MEMORY.md`（仅用户偏好/严重错误/跨会话原则）

**格式**：`## HH:MM 触发类型` + 内容（1-3 行）

## 技术原则

1. **脚本优先** — 能脚本化的不手动
2. **说做绑定** — 工具调用附带结果验证
3. **配置校验** — 修改前后必校验
4. **根因分析** — 先定位根因再修复
5. **持续演进** — 每日复盘→规则演进→Git tag

## 速率限制

- API 调用间隔 ≥ 5 秒
- 搜索间隔 ≥ 10 秒，最多 5 次/批，休息 2 分钟
- 429 错误：停止，等 5 分钟，重试
- 默认模型：根据宿主 Agent 配置

## 质量指标

### Trace 格式

```jsonl
{"timestamp":"2026-04-20T13:36:00+08:00","user_msg":"用户问题","reply":"Agent 回复","metrics":{"root_cause":{"passed":true},"essence":{"missing":false},"da_quality":"high","relevance":1.0,"consistency":true},"engine_status":"PASSED","hypotheses":["H1200-v3"],"sources":["theneuralbase 2026-04"],"da":"决策依据内容（≥50 字符）"}
```

### 质量判定

```python
PASSED = (root_cause.passed == true) AND 
         (essence.missing == false) AND 
         (da_quality == "high") AND 
         (len(da) >= 50)

FAILED = NOT PASSED

# 拦截机制（H258）
if len(da) < 50:
    engine_status = "FAILED"  # 自动拦截
```

## H608-R2 强制绑定（生效中）

**规则**：`root_cause passed=false` → `engine_status` 不能为 PASSED（自动降级为 PARTIAL）

**根因**：防止"底线通过但根因缺失"的误判

## H590-R1 RLCER 短回复容错（生效中）

**规则**：回复<200 字时，评分门槛降低（不天然低分）

**实现**：RLCER 评分增加 length_factor 归一化
