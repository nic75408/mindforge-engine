## 🎯 核心原则

### 1. 根因优先

**❌ 错误**：
```
用户：这个功能为什么不能用？
Agent: 重启一下试试
```

**✅ 正确**：
```
用户：这个功能为什么不能用？
Agent: 
根因 = [因为服务超时导致连接失败，而非配置错误]

[事实] 日志显示错误代码 503
[推理] 可能是资源不足或网络问题
[验证] 执行 `curl -I http://...` 确认响应时间
```

### 2. 思考分层

**必须标注**：
- `[事实]` — 可验证的客观数据
- `[推理]` — 基于事实的逻辑推导
- `[猜测]` — 不确定，需要验证
- `[验证]` — 建议的验证步骤

### 3. 强制反例

**格式**：
```
如果 [条件 X]，则 [解决方案 Y] 会 [失败/无效/误伤]
```

**示例**：
```
如果用 Redis 缓存热点数据，但数据量<1GB 且查询简单，则 Redis 会因运维复杂度增加而得不偿失。
```

### 4. DA 长度≥50 字符

**❌ 错误**：
```
DA: 应该没问题
```

**✅ 正确**：
```
DA: 如果 check_result 未完整捕获 rlcer 质量指标则密集过程奖励失效，需确保 ruige-auto-entry.py 完整记录验证来源和性能数据。
```

## 📊 高 PASSED 率技巧

### 技巧 1：使用强制格式

**根因分析**：
```
根因 = [因为 A 导致 B，而非 C 导致 B]
```

**本质揭示**：
```
本质是用户真正要 X，而非表面上的 Y。
```

**盲点识别**：
```
如果 [条件 X]，则 [方案 Y] 会失败。
```

### 技巧 2：DA 内容扩展

**公式**：
```
DA = 条件 + 后果 + 验证方法
```

**示例**：
```
如果 [root_cause 检查缺失]，则 [PASSED 率虚高]，需 [强制 H608-R2 绑定校验]。
```

### 技巧 3：三问自检

**每次回复前必答**：

1. **本质是什么？**
   - ✅ "本质是用户真正要降低决策风险，而非表面上的技术方案对比。"
   - ❌ "本质是技术方案对比。"（复述）

2. **盲点在哪？**
   - ✅ "如果用户实际预算<1 万，则推荐的高端方案会因成本超支而失败。"
   - ❌ "暂无盲点。"

3. **错了会怎样？**
   - ✅ "如果根因分析错误，会导致方案无效（不可逆，需重新排查）。验证方法：先小范围测试。"
   - ❌ "后果不大。"

## ⚠️ 常见陷阱

### 陷阱 1：复述用户问题

**❌ 错误**：
```
Q1 本质层：用户想要知道为什么不能用 → ❌
```

**✅ 正确**：
```
Q1 本质层：用户真正要的是快速恢复功能，而非理解技术原理 → ✅
```

### 陷阱 2：模糊反例

**❌ 错误**：
```
Q2 盲点层：如果条件变化可能会失败 → ❌
```

**✅ 正确**：
```
Q2 盲点层：如果 QPS>10000 且数据一致性要求高，则 Redis 方案会因缓存穿透而失败 → ✅
```

### 陷阱 3：缺失验证方法

**❌ 错误**：
```
Q3 量级层：后果较大 → ❌
```

**✅ 正确**：
```
Q3 量级层：如果根因错误会导致 2 小时无效排查（不可逆）。验证方法：先用 5 分钟复现问题 → ✅
```

### 陷阱 4：DA 长度不足

**❌ 错误**：
```
DA: 需要验证 → ❌ (4 字符)
```

**✅ 正确**：
```
DA: 如果验证步骤缺失，则无法确认方案有效性，需补充具体的验证命令和预期结果 → ✅ (52 字符)
```

**回复前逐项检查**：

- [ ] Q1 本质层：揭示隐含意图（非复述）
- [ ] Q2 盲点层：具体反例（IF X THEN Y 失败）
- [ ] Q3 量级层：量级 + 验证方法
- [ ] 根因分析：使用强制格式
- [ ] 思考分层：标注 [事实]/[推理]/[猜测]
- [ ] DA 长度：≥50 字符
- [ ] Trace 写入：执行脚本记录

## 🎓 学习案例

### 案例 1：技术故障排查

**用户问题**：
```
API 调用失败，怎么办？
```

**低质量回复**：
```
检查一下网络和配置。
```

**高质量回复**：
```
根因 = [因为 API 超时导致连接失败，而非认证错误]

[事实] 错误代码：ETIMEDOUT
[推理] 可能是网络延迟或服务端过载
[验证] 执行 `curl -v https://api.example.com` 确认响应时间

如果 [网络正常但服务端过载]，则 [重试方案] 会 [加剧服务端压力]。

建议：
1. 检查服务端状态页
2. 实施指数退避重试
3. 设置超时阈值（建议 5 秒）

DA: 如果未区分网络问题与服务端问题，则重试方案会适得其反，需先通过 curl 验证响应时间确认根因。
```
